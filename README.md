## The Problem

Every AI coding tool has its own config format:

| Tool | Config File |
|------|-------------|
| Claude Code | `CLAUDE.md` |
| Cursor | `.cursorrules` |
| GitHub Copilot | `.github/copilot-instructions.md` |
| Windsurf | `.windsurfrules` |
| Cline | `.clinerules` |
| Aider | `.aider.conf.yml` |

You end up maintaining **6+ config files** with the same rules. When you update one, the others go stale. Your team argues about which tool's config is "the truth."

## The Solution

**dotai** gives you a single source of truth:

```yaml
# .ai/config.yml
version: 1
project:
  name: myapp
  language: python
  framework: fastapi

rules:
  - Write clean, readable code with meaningful names
  - Add type hints to all function signatures
  - Write tests for new functionality
  - Use async/await for I/O operations

style:
  formatting: black
  linting: ruff
  line_length: 88

testing:
  framework: pytest
  coverage_target: 90
```

One command syncs it everywhere:

```bash
dotai sync
```

That's it. Every tool gets the same rules, formatted for its own config format.

## Install

```bash
pip install dotai
```

## Quick Start

```bash
# Initialize — auto-detects your project
dotai init

# Sync to all AI tools
dotai sync

# Check what exists
dotai status

# Add a rule
dotai add "Always use dependency injection"

# See what would change
dotai diff
```

## Commands

| Command | Description |
|---------|-------------|
| `dotai init` | Create `.ai/config.yml` with auto-detected settings |
| `dotai sync` | Sync config to all AI tool config files |
| `dotai sync --tool claude` | Sync to a specific tool only |
| `dotai status` | Show which tool configs exist |
| `dotai add "rule"` | Add a rule to the config |
| `dotai remove "rule"` | Remove a rule |
| `dotai rules` | List all rules |
| `dotai diff` | Preview what sync would change |
| `dotai tools` | List all supported tools |
| `dotai import claude` | Import rules from an existing tool config |

## How It Works

```
.ai/config.yml          (single source of truth)
       │
       ├── CLAUDE.md                 (Claude Code)
       ├── .cursorrules              (Cursor)
       ├── .github/copilot-instructions.md  (GitHub Copilot)
       ├── .windsurfrules            (Windsurf)
       ├── .clinerules               (Cline)
       └── .aider.conf.yml           (Aider)
```

Each tool gets its rules in its native format. Claude Code gets Markdown. Aider gets YAML. Cursor gets plain text rules. All from the same source.

## Auto-Detection

`dotai init` automatically detects:

- **Language**: Python, JavaScript, Go, Rust, Ruby
- **Framework**: Django, FastAPI, Flask, Next.js, React, Vue, Express
- **Formatter**: Black, Ruff, Prettier
- **Linter**: Ruff, Flake8, ESLint
- **Test framework**: pytest, Jest
- **Project name**: from `pyproject.toml` or `package.json`

## Why dotai?

- **DRY**: Define rules once, not six times
- **Consistent**: Every tool sees the same rules
- **Auto-detect**: Picks up your language, framework, and tools
- **Extensible**: Easy to add new tool adapters
- **Zero lock-in**: Standard YAML config, generates standard files
- **Git-friendly**: Commit `.ai/config.yml`, gitignore generated files

## Add to .gitignore

Since tool configs are generated, you may want to gitignore them:

```gitignore
# Generated by dotai — edit .ai/config.yml instead
CLAUDE.md
.cursorrules
.windsurfrules
.clinerules
.aider.conf.yml
# Keep .github/copilot-instructions.md if you want Copilot to work on GitHub
```

Or commit them — your choice. Either way, `.ai/config.yml` is the source of truth.
